# CodeCoachAI Release Acceptance Remediation and Deployment Design

## Goal

Close the release-blocking defects and the actionable correctness, security, and resource-governance risks found during the July 11 acceptance test, then deploy the resulting frontend, backend, database migrations, and Nacos configuration to the test server with a verified rollback path.

## Scope

This release covers:

- Gateway CORS and route completeness.
- Authentication failure semantics after logout or token revocation.
- Job requirement, evidence, next-action, and readiness contract alignment.
- Resume suggestion batch undo correctness.
- Resume template and artifact integrity validation.
- CSV and ICS duplicate handling, experiment assignment concurrency, and attribution enum alignment.
- Interview comparison ownership, scenario/rubric validation, and voice resource lifecycle.
- V4_058 through V4_068 database migrations.
- Full local verification, server backup, deployment, and browser/API regression.

This release does not introduce a new commercial feature, rebuild the infrastructure stack, or replace the current deployment model. Multi-instance provider orchestration will be hardened for explicit failure and cleanup; a new distributed media-provider platform is outside this release.

## Current Deployment

The test server uses Docker Compose under `/opt/codecoachai`.

- Backend artifacts: `/opt/codecoachai/jars/*.jar`
- Frontend files: `/opt/codecoachai/web`
- Compose file: `/opt/codecoachai/docker-compose.yml`
- Frontend container: `codecoachai-frontend`, exposed on port `30080`
- Gateway container: `codecoachai-gateway`, bound to `127.0.0.1:8080`
- Infrastructure: MySQL 8, Redis 7, Nacos 2.3.1, RocketMQ, Elasticsearch, and Qdrant
- Application services run as separate Java containers mounting `/app/app.jar`

The server has enough disk and memory for a backup-and-replace deployment. The permitted interruption window is five to ten minutes.

## Remediation Architecture

### 1. Gateway and Configuration

`docs/nacos` is the authoritative configuration source. Any matching file under `config/nacos` remains synchronized for manual-reference compatibility.

The Gateway route contract will explicitly include:

- Resume suggestion, template, export, artifact, experiment v2, calendar, and import endpoints.
- Interview scenarios, remediation, comparisons, TTS, and Streaming ASR endpoints.

The CORS allowlist will include the actual test origin:

```text
http://nqx.githubpage.com:30080
```

Configuration contract tests will assert that every frontend V4 API prefix is represented by a Gateway route and that the test origin is present in both supported CORS configuration forms.

### 2. Authentication Semantics

A revoked, expired, malformed, or unknown token must not cause HTTP 500.

- Browser login and logout must work from the test origin.
- A request using a revoked token must return the existing unauthenticated/token-invalid contract.
- Frontend cleanup remains unconditional if server-side logout fails.
- Authentication logs must not include token values.

### 3. Requirement and Readiness Contract

The wire contract remains backend-owned:

- Evidence uses `evidenceType`, `evidenceId`, `evidenceSubId`, `title`, `excerpt`, `resultSource`, `confidenceLevel`, `fallback`, `confirmed`, and `occurredAt`.
- Requirement actions use `actionCode`, `title`, and `path`.
- Readiness dimensions use `dimension`, `score`, `sampleCount`, `confidenceLevel`, and `fallback`.

Frontend normalization will preserve these fields and map legacy aliases only as fallbacks. It must not replace backend actions with an empty list or infer every evidence item as project evidence.

### 4. Resume Correctness and Artifact Integrity

Batch suggestion undo will continue from the latest version generated by earlier undo decisions in the same batch. Offset calculation will include only suggestions that remain accepted.

Artifact delivery will verify integrity at the download boundary:

- PDF and DOCX response bytes must match the stored artifact hash.
- Dynamically rebuilt ZIP bytes must receive a freshly calculated hash.
- ZIP manifest hashes and sizes must match actual entries.
- Template JSON must match `definitionHash` before rendering.
- Integrity failures use a stable export/artifact error instead of returning corrupt content.

### 5. CSV, ICS, Experiment, and CRM Correctness

CSV duplicate behavior:

- `SKIP` uses database atomicity for the import fingerprint.
- Only the fingerprint unique constraint maps to `CONCURRENT_DUPLICATE`.
- `CREATE` leaves the fingerprint null and permits intentional similar applications.
- The fingerprint will align with the documented seven-day duplicate rule.

Soft-delete uniqueness will be changed by a new migration rather than modifying an already deployed migration. The active-row uniqueness design must allow repeated import/delete cycles without collisions.

Experiment assignment will treat a concurrent duplicate insert as idempotent and return the existing assignment. Frontend and backend attribution metrics will share one enum set. ICS duplicate UID races will return a skipped-duplicate result rather than a generic insert error.

### 6. Interview and Voice Safety

Interview comparison queries will include the current user in the database predicate and return the same unavailable contract for missing and foreign reports.

Scenario publication will validate:

- Stage codes are unique.
- Question and time budgets are positive and internally consistent.
- Follow-up limits are valid.
- Rubric dimensions are unique.
- Rubric weights total 100.
- Scenario and rubric versions are immutable after publication.

Voice behavior:

- Provider selection comes from deploy-time configuration, with `MOCK` available only when explicitly enabled.
- The frontend derives ASR encoding from the selected `MediaRecorder.mimeType`.
- Unsupported media types fail before upload with a text fallback.
- Expired ASR sessions are cleaned by a scheduled task as well as opportunistically.
- Multi-instance or restart loss returns an explicit recoverable status and cleanup instruction.
- Component teardown synchronously stops local tracks, recorder, timers, animation frames, audio contexts, and object URLs; remote cancellation remains bounded and idempotent.
- Audio deletion reconciliation can move a timed-out/cancelled record to deleted when the remote file is confirmed absent.

## Database Migration Strategy

Before deployment:

1. Back up the `codecoachai_v1` database.
2. Read Flyway history and record checksums.
3. Confirm the environment state of V4_058 through V4_068.
4. Never modify a migration already recorded in Flyway history.
5. Add a later migration for any unique-index correction required by this remediation.
6. Run Flyway validate.
7. Execute pending migrations once through the deployment migration path.
8. Verify tables, columns, indexes, character set, and Flyway history.

If validation fails, deployment stops before replacing application services.

## Verification Strategy

### Local gates

- Targeted red/green tests for every defect.
- Backend module tests for resume, interview, auth/security, Gateway configuration, and AI integration.
- Backend full compile, full test, and package.
- Frontend type-check, full unit tests, contract/quality gates, and production build.
- Migration numbering, duplicate-index, and checksum checks.
- `git diff --check`.

### Server gates

- Compose configuration validates.
- Nacos contains the intended Gateway and service configuration.
- All application containers start without restart loops.
- All services register in Nacos.
- Gateway health and routing checks pass.
- Flyway history contains the expected migrations.
- Browser login/logout works for administrator and normal user.
- Network has no unexpected 400, 401, 403, 404, 409, or 500 responses.
- Four-priority acceptance flows are re-executed.

## Deployment Procedure

### Backup

Create a timestamped release directory containing:

- Current JAR files.
- Current frontend directory.
- Current Compose file.
- Exported Nacos configuration.
- MySQL dump.
- Container image, status, and health inventory.

No existing backup is overwritten.

### Artifact staging

Upload new artifacts into a timestamped staging directory. Verify local and remote SHA-256 values before activation.

### Activation order

1. Apply validated database migrations.
2. Publish Nacos configuration.
3. Replace and restart `resume`, `interview`, and `ai`.
4. Replace any affected auth/security service.
5. Replace and restart Gateway.
6. Replace frontend files through an atomic directory switch or container recreation using the staged directory.
7. Restart only affected supporting services when configuration refresh cannot be performed safely.

Infrastructure containers are not recreated.

### Rollback

Rollback is triggered by migration validation failure, container restart loops, failed health checks, browser login failure, or a core-flow regression.

- Restore prior JARs and frontend directory.
- Restore prior Nacos configuration.
- Recreate affected application containers.
- Database restoration is used only when a migration changed data/schema incompatibly and application rollback cannot operate against the new schema.
- Preserve failed-release logs and artifacts for diagnosis.

## Security

- Server, database, Nacos, AI, OSS, and application secrets remain outside version control.
- Deployment commands do not print credentials or token values.
- Remote backups have root-only permissions.
- `/inner/**` routes remain absent from the public Gateway.
- Test user data is not deleted.
- No unrestricted load test is run.

## Acceptance Decision

The release is considered successful only when:

- All P0 defects are closed.
- No unresolved P1 causes data corruption, security exposure, or unrecoverable failure.
- Local full gates pass.
- Server deployment and migrations pass.
- Administrator and normal-user browser flows pass.
- The updated acceptance report explicitly allows continued testing and states the gray/production decision.
