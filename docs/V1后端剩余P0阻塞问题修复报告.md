# V1 后端剩余 P0 阻塞问题修复报告

## 1. 本轮只修复的两个问题

本轮只处理两个 V1 P0 阻塞项：

- 用户端题库标签筛选失败。
- 面试报告仍出现英文 `Mock report` 兜底内容。

未新增独立接口，未修改前端仓库，未实现 V2/V3 功能。

## 2. 标签筛选根因

用户端 `/questions` 列表原来只返回 `tags: ["E2E_TEST_缓存"]` 这种标签名称数组，没有返回真实标签 ID。

前端只能根据名称生成临时伪 ID，再把伪 ID 作为 `tagId` 请求后端。后端实际按 `question_tag_relation.tag_id` 精确过滤，因此伪 ID 与真实 `tagId=100002` 不匹配，导致筛选结果为 0。

## 3. 后端返回标签 ID 的实现方式

本轮在现有 `/questions` 和 `/questions/{id}` 响应中补充真实标签 ID，不新增接口：

- `tags` 改为结构化标签数组，元素包含 `id`、`name`、`tagName`、`status`。
- 同时补充 `tagIds`，用于前端直接拿真实 ID 发起 `tagId` 筛选。
- 同时补充 `tagNames`，用于前端继续做标签名称展示。

示例结构：

```json
{
  "tags": [
    {
      "id": 100002,
      "name": "E2E_TEST_缓存",
      "tagName": "E2E_TEST_缓存",
      "status": 1
    }
  ],
  "tagIds": [100002],
  "tagNames": ["E2E_TEST_缓存"]
}
```

用户端 `/questions` 仍然强制叠加 `status=1`，管理端 `/admin/questions` 原有 `tagId/status` 筛选逻辑未破坏。

## 4. Mock report 根因

代码中 `codecoachai-ai` 的 `AiServiceImpl.generateReport` 仍硬编码英文兜底：

- `Mock report: the interview has been completed...`
- `Shows basic understanding...`
- `Needs more depth...`
- `Review JVM...`

浏览器报告页出现英文，可能来自两类来源：

- 已生成并保存在数据库 `interview_report` 表中的历史英文报告。
- 新完成面试时调用 AI mock 服务生成的英文报告。

本轮已处理两条路径：

- `codecoachai-ai` 的报告生成兜底改为中文结构化内容。
- `codecoachai-interview` 查询已有报告时，如果识别到旧英文 Mock report，会自动替换为中文兜底报告并保存，避免数据库未重新导入时继续阻塞演示。

## 5. 是否需要重新导入 sql/dev_test_data.sql

不强制需要重新导入。

原因：

- `sql/dev_test_data.sql` 当前已经没有英文 `Mock report` 种子内容。
- 对于数据库中已经存在的旧英文报告，本轮代码会在查询报告时自动中文化并回写。

如果希望把本地演示数据整体恢复到最新中文样例，可以重新导入：

```bash
mysql -u root -p codecoachai_v1 < sql/dev_test_data.sql
```

实际账号、密码和 MySQL 路径以本机环境为准。
